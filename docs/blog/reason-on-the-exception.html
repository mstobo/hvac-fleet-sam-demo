A three-sensor demo that spent $500 in a day

We recently saw a familiar pattern on a small industrial dashboard—three pumps, one bearing-temperature stream, event broker connected, simulator running. Not enterprise scale. Not big data.

The design treated live telemetry like chat input: each forwarded reading—and retries when tools failed— triggered another LLM pass. The UI stacked three columns: live events, AI analysis, recommended actions. The analysis column accumulated hundreds of identical tool errors while the actions column still produced long prescriptive text (shift plans, downtime cost ranges, approval chains).

On the order of 1,500 telemetry events and 1,100+ analysis attempts in one day—for three sensors. The gateway meter showed about $500 in token spend in 24 hours. That is not a spreadsheet projection; it is what happens when the stream hits the model.

Spend was capped when LiteLLM gateway controls cut off the runaway path. That is a circuit breaker, not architecture. The fix is upstream: never wire IIoT throughput to the reasoning engine—keep AI off the hot path.

This reference demo exists to show the alternative—deadband, sketches, and rules on the broker path; agents only on exceptions.

The uncomfortable truth

Many IIoT AI pilots do not fail on model quality. They fail when finance asks what you are paying per sensor per day after real tag counts show up.

A point publishing every two seconds generates on the order of 43,000 messages per sensor per day. If each message becomes a few hundred tokens through an LLM, pilot economics (ten sensors) hide production economics (hundreds or thousands of sensors). You are paying reasoning rates to hear normal, normal, normal most of the time.

That is not a model problem. It is an architecture problem.

LLMs are reasoning engines, not stream processors

The natural pilot diagram is Sensor → LLM → Insight. It works with a handful of tags. It breaks at scale.

Correct pattern: keep filtering and summarization on the broker-centric data plane; invoke AI when operators ask—or when a high-value exception (e.g. fleet-critical) fires.

Cheap (data plane)Expensive (reasoning plane)
Suppress insignificant changeCorrelate patterns across assets
Thresholds and zonesRank probable causes
Rolling window statsOperator-ready narrative
Deterministic alertsRecommendations under uncertainty

Sketches, SoT, and where chain-of-thought belongs

Industrial AI needs two compression patterns on two planes. Conflating them is how a three-sensor demo spends $500 in a day.

Sketch-of-Thought on the stream (data plane)

A sketch is deterministic operator language (Python templates, zero generative cost on the stream)—zone, delta %, 30s stats, severity flags.

Sketch-of-Thought (Aytes et al., 2025) shortens how a model writes its reasoning trace. Our sketches shorten what the model reads from telemetry—same compress-the-story instinct, but deterministic ingress, not SoT routing on every MQTT message.

Chain-of-thought on the exception (reasoning plane)

Chain-of-thought prompting (Wei et al., 2022) fits when you need hypotheses, timelines, and ranked actions—once per incident, not per 2-second publish. In this demo, CoT appears only when SAM runs:

  • Curated inputs — sketches, chart URLs, and alerts from tools, not the live firehose.
  • Mandated tool order — incident context first, charts on the same UTC window, dispatch last (SECTION A budget).
  • Structured output — sections 1–8 so reasoning becomes operator prose, not an unbounded hidden monologue.
  • Multi-turn tool loop — several LLM turns to call tools and assemble the report; CoT orchestrated by the gateway, not CoT on every reading.
PlanePatternCompressesLLM cost
Data planeSoT-inspired sketches (optional jargon)Telemetry → narrative before the agent0 on ingest

The $500 anti-pattern is chain-of-thought on the stream. This stack front-loads SoT-style sketches and reserves CoT for fleet-critical and operator deep-dives.

Natural language

machine-002:motor_temp_c recorded a 6.2% spike to 80.4°C.
30s window: mean 76.1°C, range [74–82°C]. Zone: CRITICAL.

Jargon (Expert-Lexicon style)

m2:mot Δ↑6.2% T80.4 Z:C μ30=76.1[74-82] !CRIT

Toggle NL / Jargon on the live dashboard or set SKETCH_STYLE=jargon. Offline A/B: tools/sketch-token-lab.

Case study: automated fleet analysis (SECTION A)

When correlated fleet-critical fires (default ≥50% of active assets in CRITICAL), one analysis-request triggers FleetQueryAgent with a hard tool budget: three machine-level incident contexts, three combined machine charts, no per-point forensic loop, one dispatch call. Output: sections 1–8 once, three machine-plotly-html URLs under Chart Evidence.

Replay, queues, and stale reports

Broker replay does not automatically cut SAM cost. Re-analyzing the same incident runs the LLM again unless you dedupe on correlation_id.

What works: store analysis-response as an incident snapshot and republish on replay—point-in-time narrative for audit, not live ops.

Production habits

  1. AI off the IIoT hot path — filter, sketch, and rule-detect before any model sees telemetry.
  2. Tool budgets on automation — SECTION A is the template.
  3. Measure one change at a time — Slack token footers on repeated FLEET_CRITICAL runs.
