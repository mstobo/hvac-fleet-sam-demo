mqtt5SRDemo — MQTT5 HVAC-style pipeline

Deterministic data plane on Solace · optional SAM / LLM query plane · SQLite as the curated store

PROBLEM → APPROACH

High-frequency MQTT cannot be the direct input stream to an LLM. This repo implements a filter → summarize → detect chain on the broker path so downstream consumers (dashboards, CMMS, agents) see low-noise, human-readable events and stored history.

KEY NUMBERS (FROM README NARRATIVE)

  • Deadband + heartbeat suppresses the bulk of unchanged readings.
  • Sketches are template NL, not generative AI.
  • Anomaly stage is rule-based (zones, spikes, fleet fraction).
  • LLM / SAM sits on the query path over SQLite, not per reading.

Control plane, clients & governance

How optional AI, the web twin, Java samples, and Event Portal fit around the same broker

QUERY PLANE

Solace Agent Mesh (sam run) can expose tools that read SQLite (sketches, alerts, fleet status) so natural-language questions do not replay the raw topic firehose. Can also respond to event-driven prompts through Solace event gateway.

  • Configs under sam/configs/agents/ (e.g. fleet query, orchestrator).
  • Prescriptive YAML (sensor_pipeline_sam.yaml) documents an alternate SAM-centric view; Python microservices implement the dc/… pipeline in code.

OTHER CLIENTS IN THE REPO

  • demo_dashboard.html — Web UI + digital twin; publishes/consumes the same dc/… topics (optional °C in topic path).
  • demo_publisher.py — Continuous HVAC-style simulator (raw path without trailing °C in topic).
  • Java MQTT5 — Separate sample using MqttConfig.TOPIC_BASE (test/mqtt5/messages) with optional JSON Schema / SERDES.

EVENT PORTAL GOVERNANCE

Event types are modeled with past-tense names, topic addresses on each event version, and declared producers/consumers for deadband, sketch, anomaly, and the web dashboard on raw variants. Optional: Java and SAM workflow objects live in the same application domain for a single governance view.
